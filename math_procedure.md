# math_procedure.md — 수학 문제 시각화 풀이 절차 (고교)

> **상위 공통 절차는 `PROCEDURE.md`.** 이 파일은 **수학 문제**에 한정한 세부 절차다. (물리·광학은 별도 파일 예정)
> 대상 범위: **고교 수준** — 함수·미적분·기하·이차곡선·확률·수열·벡터 등.
> **성공 판정**: 단순히 "풀었다"가 아니라, 독자가 **"다음에 비슷한 게 나와도 안 틀리겠다"는 확신**을 얻는 것.
>   ("풀었는데 다음에 또 나오면 틀릴 것 같은 예감" — 이걸 없애는 게 이 절차의 존재 이유.)
> 이 절차는 편집장님과의 회의로 계속 개정한다. (개정 이력은 맨 아래)

---

## 왜 "문제 주도"인가
`PROCEDURE.md`의 6단계는 **개념 주도**(글 기획: 무엇을 보여줄까에서 출발).
이 절차는 **문제 하나에서 출발**한다 — 주어진 문제를 시각적으로 풀고, 마지막에 **"이 문제가 왜 어려웠는지"까지 해부**해서 유형 자체를 손에 쥐게 만든다.
AI가 가장 잘하는 건 분석이다. 그래서 "푸는 것"에서 멈추지 않고 **"왜 어려운지 분석"**까지 가는 것이 이 절차의 차별점(6단계).

---

## 절차 한눈에

```
0. 분류    문제 유형 판별 → 시각 표현 선택 (아래 유형표)
1. 재현    문제 상황을 좌표평면/그래프에 그대로 그림 (정적 도해)
2. 일반화  이 문제 = 일반 상황의 "특수한 한 값" → 슬라이더/드래그로 전체를 보임
3. 유도    WolframLanguageEvaluator로 풀이 흐름을 단계별 검증
4. 내레이션 풀이 = "그림이 변해가는 서사"로 서술
5. 정답    ★맨 마지막에 정답 명시 + 오답 선지가 왜 오답인지
6. 해부    ★왜 어려웠나 분석 + 형제 유형 + 함정 지도 (이 절차의 핵심)
```

---

## 0. 분류 (Triage)

문제에서 세 가지를 먼저 뽑는다:
- **변수** — 무엇이 움직이나 (구하려는 대상, 자유롭게 변하는 양)
- **파라미터** — 무엇이 고정인가 (주어진 상수·조건)
- **목표** — 무엇을 구하나 (넓이·개수·좌표·범위 등)

그 뒤 아래 유형표에 매핑해 **1순위 시각 표현**을 고른다.

| 문제 유형 | 대표 소재 | 1순위 시각 표현 |
|---|---|---|
| 이차곡선·도형·접선·교점 | 원/타원/포물선, 외부점 접선, 두 곡선 교점 | 좌표평면 + (필요시 축 회전 그리기) |
| 함수·미분 (증감·극값·변곡) | 다항·삼각·지수로그 함수 | 그래프 + 접선 기울기 애니메이션 |
| 적분 (넓이·부피) | 정적분, 회전체 | 넓이 채우기 / 단면 애니메이션 |
| 방정식·부등식의 해 | 실근 개수, 해의 범위, 판별식 | 두 그래프 교점 / 수직선 |
| 확률·경우의 수 | 조건부확률, 순열·조합 | 표본공간 격자 · 트리 · 반복 시뮬 |
| 수열·극한 | 점화식, 급수 수렴 | 항의 위치 애니메이션 (수렴을 눈으로) |
| 벡터·공간도형 | 내적, 평면·직선 위치관계 | 2D 사영 (입체가 꼭 필요하면 Three.js) |
| 정수·조합 논증·귀납 등 | 정수론, 수학적 귀납법 | **시각화 이득 낮음 → 구조도/표 위주, 솔직히 명시** |

> 마지막 행이 중요하다. **모든 문제가 시각화로 쉬워지지는 않는다.** 이득이 낮은 유형은 억지로 그리지 말고 "이 유형은 시각화보다 논리 전개가 핵심"이라고 정직하게 밝힌다. (제0원칙: 남 눈치용 화려함 금지)

## 1. 재현 — 문제 상황을 그대로 그린다

- 정적 도해(A단계, Rough.js/SVG 빌드타임)로 **문제에 나온 그대로**를 좌표평면·그래프에 올린다.
- 목적: "문제가 말로만 서술한 상황"을 한 장면으로 고정. 여기서 이미 절반은 보인다.
- 숨은 변환이 있으면(예: `xy`항 → 기울어진 축) **이 단계에서 드러낸다.** 표준형으로 착각하지 않도록.

## 2. 일반화 — "이 문제는 특수한 한 값"

- 핵심 철학: **주어진 문제는 어떤 일반 상황의 특수한 한 값일 뿐.**
- 인터랙티브 시뮬(B단계, React island)로 파라미터를 슬라이더/드래그로 만들어 **전체 그림에서 이 문제가 어디에 있는지**를 보인다.
  - 예: 외부점에서 곡선에 긋는 접선 문제 → 외부점을 드래그하면 접선이 **2개 → 1개(곡선 위) → 0개(내부)**로 바뀌는 전환을 눈으로. 주어진 점은 그 연속 변화 속 한 스냅샷.
- 이 단계가 "다음에 또 나와도 안 틀림"의 뿌리다 — **특정 숫자가 아니라 구조를 보게** 만든다.
- 형태·리얼리즘 목표가 정해지면 **구현 전 편집장님 확인** (작업 규칙: 독단 실행 금지).

## 3. 유도 — 단계별 검증 (`PROCEDURE.md` 3번·`CLAUDE.md` 9번과 동일)

- 풀이 유도는 `WolframLanguageEvaluator`로 **코드로 짜서 단계별(대입→전개→풀이) 실행** → 각 중간식을 커널이 검증.
- 검증된 중간식만 본문에 사용. 커널은 상태 비저장이라 매 호출에 정의를 다시 포함.

## 4. 내레이션 — 풀이 = 그림이 변해가는 서사

- 순서: **시각물이 먼저, 텍스트가 나중.**
- 풀이과정을 "식1 → 식2"가 아니라 **"이 선을 이렇게 움직이면 → 그림이 이렇게 변하고 → 그래서 식이 이렇게 된다"**로 서술.
- 결론만 던지지 않는다. "왜 그렇게 되는지"를 그림과 번갈아. 수식은 KaTeX, 3번에서 검증된 식 사용.

## 5. 정답 — 맨 마지막에

- 시각적 유도로 끝까지 끌고 간 뒤, **최종 정답을 명확히 명시한다.** (독자가 답을 확인할 수 있어야 함)
- 객관식이면 **각 오답 선지가 왜 오답인지**도 짧게 — 흔한 실수가 어떤 오답으로 이어지는지 보이면 6단계와 연결된다.
- 최종 수치는 `WolframAlpha`로 한 번 더 교차확인 (독립 경로 검증).

## 6. 난이도 해부 — "다음에 안 틀리게" ★이 절차의 하이라이트

문제를 다 풀고 나서, **왜 어려웠는지**를 구조적으로 뜯는다. 세 가지를 반드시 작성:

**(a) 개념 혼합 지도** — 이 한 문제에 몇 개의 개념이 겹쳤나? 각각을 분리해 나열.
- 어려움의 정체는 대개 **"개념 하나가 어려워서"가 아니라 "여러 개념이 직렬로 엮여서"**다. 하나만 막혀도 못 풀고, 순서까지 맞아야 풀린다.
- 예 (이차곡선 외부점 접선 + 넓이 유형):
  ① 회전된 이차곡선 인식 (`xy`항 → 기울어진 타원)
  ② 외부점에서 곡선에 긋는 접선 조건 (판별식 = 0)
  ③ 접점 좌표 구하기
  ④ 세 점으로 삼각형 넓이
  → **4개 개념이 사슬처럼 연결**. 이 사슬 구조를 보여주는 것 자체가 "왜 어려운지"의 답.

**(b) 함정 지도** — 틀리기 쉬운 지점을 콕 집는다.
- **숨은 변환**: `xy`항을 못 보고 표준형 타원으로 착각
- **부호·판별식 실수**: 접선 조건에서 부호 하나로 답이 갈림
- **특수를 일반으로 착각**: 이 값에서만 성립하는 걸 공식처럼 외움
- 각 함정을 "이 함정에 빠지면 → 어떤 오답으로 가는지"까지 연결 (5단계 오답 분석과 맞물림)

**(c) 형제 유형 (유형 일반화)** — "이 문제의 형제들"과 공통 해법 뼈대.
- 같은 골격의 변형: 원/포물선 버전, 접선을 법선으로 바꾼 버전, 넓이를 길이·각도로 바꾼 버전 등
- **공통 패턴 한 줄**: 예) "외부점 접선 → 판별식 0으로 접선 확정 → 접점 → 목표량 계산." 이 뼈대만 쥐면 소재가 바뀌어도 같은 손동작.

> **6단계 성공 판정**: 독자가 "아, **이 유형은 이 뼈대구나**" 하고 다음 변형을 스스로 예측할 수 있으면 성공. 여기까지 와야 "다음에 또 나와도 안 틀림"이 완성된다.
