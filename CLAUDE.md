## 0. 제1원칙 — 독자는 오직 "나"(편집장님)
이 사이트의 독자는 편집장님 **자신 한 명**이다. 타인이 아니다.
- 이전 시도들이 무너진 이유: 타인을 독자로 상정 → 눈치·자기검열 → 지쳐서 중단.
- 따라서 **불필요**: SEO, 방문자 분석, 댓글, 공유 버튼, 친절한 설명 강박, "발행"의 부담,
  꾸준함 압박, 가독성·접근성 "표준"(참고만). 성격은 블로그보다 **"내 생각 서랍 + 놀이터"**.
- **자유**: 미완성·틀린 생각·메모도 그대로 둔다. 주제는 편집장님이 궁금하면 그게 이유.
  디자인은 편집장님 눈에 예쁘면 정답.
- **모든 결정의 잣대**: "이건 남 눈치인가, 내가 원하는가?" — 남 눈치면 버린다.
- **클로드(나)에게**: "보통 이렇게 합니다 / 모범사례 / 표준"을 이유로 편집장님 취향을 누르지 말 것.
  기술적 사실(안 되는 것·비용)은 정직히 말하되, 취향의 영역에선 편집장님이 원하는 방향을 최우선으로.

## 0-1. 제2원칙 — 시각화가 주인공, 텍스트는 내레이션
이 블로그는 텍스트가 넘치는 블로그가 아니라 **시각화(Visualization)·시뮬레이션이 중심인 블로그**다.
- **핵심 믿음**: 시각화하면 이해가 빨라지고, 텍스트만으론 안 되던 이해도 된다. (H2O를 분자식으로만 보면 안 보이던 결합각이, 3차원으로 그리면 비로소 보이는 것처럼.)
- **주객 관계**: 시각화/시뮬이 주인공, 텍스트는 그걸 안내하는 내레이션. 글을 쓰다가 그림을 곁들이는 게 아니라, **"무엇을 눈에 보이게 만들까"에서 출발**한다.
- **성공 판정 기준**: "안 보이던 게 보이게 됐는가?"
- **클로드(나)에게**: 결론만 던지지 말 것. 유도 과정을 풀어서 보여주되, 그 중심은 언제나 시각물. **글/시뮬 작업 시 `PROCEDURE.md`(공통) + 주제별 절차서(수학=`math_procedure.md` 등)를 반드시 따를 것.** (관련: 5번 시뮬 스택, 6번 손그림 톤, 9번 도구 호출 규칙)

## 5. 확정 기술 스택 (2026-07-03 회의 확정)
사이트의 성격 = **화려한 인터랙티브 시뮬레이션이 많은 개인 블로그** (첫 예시: 안구+볼록렌즈 초점/도수 시뮬).
- **뼈대**: Astro (구조·라우팅·정적 빌드)
- **글**: MDX — 마크다운 글 안에 시뮬 컴포넌트를 직접 삽입
- **인터랙티브 island**: React (`@astrojs/react`) — 시뮬을 컴포넌트로, 슬라이더·상태 관리
- **시뮬 렌더링**: **2D(광선·초점 등 광학) → Canvas/SVG**, **3D(안구 모델·투명 굴절) → Three.js(+react-three-fiber/drei)**.
  Three.js는 "기본"이 아니라 3D가 필요할 때만 꺼내는 카드. 광학 시뮬은 물리엔진 불필요(렌즈 공식=기하 계산).
- **수식**: KaTeX
- **스타일**: Tailwind (NAS choo-invest에 v4 전례, 재사용)
- **배포**: 로컬(i5)에서 정적 빌드 → NAS 정적 호스팅, 기존 cloudflared 터널에 Public Hostname만 추가.
- **접은 것**: 언리얼(웹 배포 불가), 블렌더는 폐기 아님 — 예쁜 3D 안구 모델(.glb) **제작 공방**으로만 사용.
- **리얼리즘 사다리**: 1)교과서 작도 2)수차 재현 3)정밀 광선추적(스넬) 4)임상 안구모델 5)3D+환자가 보는 상.
  단계별로 물리지식·시간 비용 증가. 시뮬마다 "물리적 정확성" vs "시각적 사실감" 목표를 따로 정하기로 함(미결).

## 9. 도구 호출 규칙 — Wolfram (2026-07-04 확정)
수학·물리·안과학·광학 콘텐츠에 계산이 들어갈 때 지키는 규칙. 연결된 울프람 도구는 성격이 다른 2종.
- **`WolframAlpha`**(자연어 질의): 최종 답만 요약해서 줌. `step by step`을 붙여도 중간 과정은 안 나옴(웹의 Step-by-step solution은 Pro 전용이라 API 미포함). **용도: 결과값 팩트체크·교차검증 전용.**
- **`WolframLanguageEvaluator`**(실제 Wolfram Language 커널 실행): 클로드가 유도 논리를 코드로 직접 짜서 단계별(대입→전개→풀이 등)로 실행시키면, 각 단계를 진짜 계산 엔진이 검증해줌. **용도: 풀이과정 자체를 신뢰도 있게 구성.** (도구 자체는 상태 비저장 커널이라 이전 계산 재사용 불가, 매 호출에 필요한 정의를 다시 포함)
- **원칙(0-1 목표와 직결)**: 답만 주지 않는다. 수식이 필요한 글은 `WolframLanguageEvaluator`로 유도 흐름을 단계별 실행 → 검증된 중간식을 그대로 서술형 설명에 사용 → 최종 수치는 `WolframAlpha`로 한 번 더 대조.
- **시각자료 원칙: 정확성 > 톤** (2026-07-04 개정). 시뮬·그래프는 정확해야 하며, 울프람 산출 이미지도 케이스별로 사용 가능(검증·참고용 자유, 게재용은 그대로 쓸지 톤에 맞게 다시 그릴지 편집장님과 결정). 손그림 톤(6번)은 장식·도해 영역에 적용. 상세는 `PROCEDURE.md` 4번.
