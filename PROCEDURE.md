## 절차 한눈에

```
1. 출발점     "무엇을 눈에 보이게 만들까" 정의   ← 텍스트 초안부터 쓰지 않는다
2. 형태 결정   정적 도해 → 인터랙티브 → 3D 중 필요 최소 단계 + 리얼리즘 사다리 목표
3. 수식·유도   WolframLanguageEvaluator로 단계별 실행·검증
4. 시각물 제작  정확성 > 톤. Rough.js/SVG/Canvas/Three.js, 울프람 이미지도 케이스별 사용
5. 내레이션    시각물을 안내하는 텍스트 작성 (시각물이 먼저, 글이 나중)
6. 검증       WolframAlpha 교차확인 + 시뮬 극단값 테스트
```

## 1. 출발점 — "무엇을 보여줄 것인가"

글 기획은 텍스트 개요가 아니라 **시각화 목표 한 문장**으로 시작한다.

- 형식: *"[개념]에서 [텍스트만으론 안 보이는 것]을 [이런 시각물]로 보이게 만든다."*
  - 예: "난시에서 **두 초선(전초선·후초선)이 왜 생기는지**를, 토릭 렌즈의 두 경선을 따로 굴절시키는 시뮬로 보이게 만든다."
- 이 한 문장이 안 나오면 아직 글 쓸 때가 아니다 — 주제를 더 파거나 편집장님과 회의.
- 한 글에 시각화 목표는 1~2개까지. 많으면 글을 쪼갠다.

## 2. 형태 결정 — 필요 최소 단계

시각물 형태는 아래 사다리에서 **목표를 달성하는 가장 낮은 단계**를 고른다. (화려함이 목적이 아니라 "보이게 만드는 것"이 목적)

| 단계 | 형태 | 도구 | 언제 |
|---|---|---|---|
| A | 정적 도해 | Rough.js/SVG (빌드타임) | 구조·관계가 한 장면으로 보일 때 |
| B | 인터랙티브 시뮬 | React island + Canvas/SVG | 변수를 움직여봐야 보일 때 (슬라이더 등) |
| C | 3D | Three.js (r3f/drei) | 3차원이어야만 보일 때 (안구 모델, 입체 굴절) |

- 함께 정할 것: **리얼리즘 사다리 목표** (CLAUDE.md 5번: 교과서 작도 ~ 임상 모델) — 시뮬마다 "물리적 정확성 vs 시각적 사실감" 중 무게를 명시.
- 형태·목표가 정해지면 **구현 전에 편집장님 확인** (작업 규칙 7번: 독단 실행 금지).

## 3. 수식·유도 — 단계별 검증 (CLAUDE.md 9번과 연동)

- 유도가 들어가는 글은 `WolframLanguageEvaluator`로 **유도 흐름을 코드로 짜서 단계별(대입→전개→풀이) 실행** → 각 중간식을 커널이 검증.
- 검증된 중간식만 본문 서술에 사용. 검증 안 된 식을 "아마 이럴 것"으로 쓰지 않는다.
- 커널은 상태 비저장 — 매 호출에 필요한 정의를 다시 포함.
- 시뮬 코드에 들어가는 공식(예: 얇은 렌즈 방정식의 부호 규약)도 구현 전에 같은 방식으로 검증.

## 4. 시각물 제작 — 정확성 우선, 톤은 그 다음

- **원칙 서열: 정확성 > 톤.** 시뮬레이션·그래프는 물리적·수학적으로 정확해야 한다. 손그림 톤 때문에 정확성을 희생하지 않는다.
- 톤의 적용 범위: 손그림(Rough.js seed 고정, 분필/색연필 팔레트, Gaegu 라벨)은 장식·구조 도해·UI 프레임에. 데이터가 실리는 곡선·광선·축은 계산 결과 그대로 정확하게 (Rough.js풍으로 그리더라도 좌표는 정확히).
- **울프람 이미지는 무조건 미사용이 아니다.** 케이스별 판단:
  - 검증·참고용(내부 확인)은 자유롭게 사용
  - 정확성이 중요한 그래프는 게재용으로도 사용 가능 — 그대로 쓸지, 톤에 맞게 다시 그릴지는 그때그때 편집장님과 결정
- 도구: Rough.js·SVG·Canvas·Three.js(+울프람 산출물). 시뮬은 React island로 MDX 글 안에 삽입 (독립 전시물이 아니라 글의 설명 도구).
- 좌표·곡선 등 그리기용 수치 계산은 로컬(파이썬/JS)로 해도 됨 — 검증이 필요한 건 물리·수학 공식이지 픽셀 좌표가 아니다.

## 5. 내레이션 — 시각물을 안내하는 텍스트

- 순서: **시각물(또는 그 설계)이 먼저, 텍스트가 나중.** 텍스트는 "그림을 보면서 따라오는" 구조로.
- 톤: 독자는 편집장님 한 명 (제1원칙). 친절 강박·SEO·표준 불필요. 미완성 표시·낙서 OK.
- 결론만 쓰지 않는다 — "왜 그렇게 되는지"를 시각물과 번갈아 서술.
- 수식은 KaTeX. 3번에서 검증된 중간식을 그대로 사용.

## 6. 검증 — 배포 전 마지막 관문

- **수치**: 최종 수치·결과값은 `WolframAlpha`로 한 번 더 교차확인 (3번과 별개의 독립 경로).
- **시뮬**: 극단값 테스트 — 슬라이더 양 끝, 0, 음수, 특이점(초점거리 무한대 등)에서 깨지지 않는지.
- **화면**: `npm run build` 통과 + 실제 화면에서 시각물이 의도대로 보이는지 확인.
- **최종 판정**: "안 보이던 게 보이게 됐는가?" — 아니라면 2번(형태)부터 다시.
